# Строгая валидация bulk fill и integer input

**Date:** 2026-06-23 14:07
**Files:** `lib/atdbBatchEdit.ts`, `components/BulkEditDialog.tsx`, `scripts/check-atdb-batch-edit.mjs`
**Severity:** high

## Problem

Bulk operation `fill` without an explicit value was accepted as valid and behaved like a clear operation. Numeric UI input also used partial integer parsing, so strings such as `12abc`, `1e2`, or `12.9` could be converted into another integer instead of being rejected.

## Root Cause

`normalizeOperationValue()` intentionally mapped `null` and `undefined` to `null` for clear semantics, but the validation path reused it for `fill` without distinguishing the action. The dialog used `Number.parseInt`, which accepts numeric prefixes instead of validating the complete input string.

## Solution

`fill` now rejects `null` and `undefined` at the batch operation boundary, while `clear` remains the only action that writes `null`. A shared strict integer input parser validates the full string and safe integer range before conversion, and the dialog uses that parser for numeric and place-link controls. The batch regression script now covers missing fill values and partial numeric inputs.

## Prevention

- Validate runtime action semantics separately from shared value normalization helpers.
- Do not use `parseInt` for user-facing integer validation unless prefix parsing is explicitly desired.
- Add regression cases for missing values, `null`, partial numeric strings, exponential notation, and decimals whenever adding bulk-edit or CLI input parsers.
