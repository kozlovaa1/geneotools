# Restore parseAtdb compatibility export

**Date:** 2026-05-27 18:23
**Files:** lib/parseAtdb.ts
**Severity:** medium

## Problem

Verification found that `lib/parseAtdb.ts` no longer exported the runtime `parseAtdb` function, even though the decomposition plan required this compatibility entrypoint to keep working.

## Root Cause

The file remained as a type-only compatibility layer during the `.atdb` processor decomposition. Because current app imports use `lib/sqlProcessor.ts`, the missing runtime export was not caught by normal project compilation.

## Solution

Added a direct re-export from `lib/parseAtdb.ts` to `lib/sqlProcessor.ts`:

```typescript
export { parseAtdb } from './sqlProcessor';
```

This restores the compatibility contract without changing parsing behavior or adding runtime logging.

## Prevention

- Include compatibility entrypoints in verification import checks after facade refactors.
- When a plan names a compatibility file, verify both type exports and runtime exports.
- Add a focused compile check or unit test that imports `parseAtdb` from `lib/parseAtdb`.
