# Полное применение ATDB mapping в readers и write-safety

**Date:** 2026-06-04 20:28
**Files:** lib/atdb/mapping.json, lib/atdb/mapping.ts, lib/atdb/mappingTypes.ts, lib/atdb/schemaContext.ts, lib/atdb/readers/eventsReader.ts, lib/atdb/readers/personsReader.ts, lib/atdb/writers/valueWriter.ts, scripts/check-atdb-mapping.mjs, scripts/check-atdb-write-safety.mjs
**Severity:** high

## Problem

Readers интерпретировали первое найденное значение даты или ссылки события без проверки `f_id`. Родительские роли зависели от фиксированных ID. Точечная запись `ValuesLinks` удаляла все ссылки с совпадающими `f_id`, `rec_table` и `rec_id`, не ограничивая удаление принадлежащим writer `vlink_table`. Mapping-gate проверял fixtures в основном по `rec_table`.

## Root Cause

Канонический mapping был подключён к общему resolver, но не применён до конечных SQL-предикатов и каталоговой семантики ролей. Контракт владения ссылками описывал target-таблицу, однако общий write-helper не использовал её при удалении. Regression-gates проверяли общий успешный parse/build, но не создавали конфликтующие данные, способные выявить эти ошибки.

## Solution

- Добавлено invariant-правило даты события и фильтрация event values по разрешённым `f_id`.
- Сохранено терпимое чтение fixture-specific альтернативной ссылки события.
- Семантика родительских ролей разрешается по `EventRoles.et_id`, `roletype` и `ord`, независимо от числового ID роли.
- Удаление `ValuesLinks` ограничено принадлежащим writer `vlink_table`.
- Write-safety regression теперь меняет ID ролей и добавляет чужой link target с тем же `f_id`.
- Mapping-gate сверяет каталоги полей, типы событий, роли, link targets и orphan-ссылки во всех доступных fixture-снимках.

## Prevention

- Любая семантика `Values*` должна подтверждаться одновременно `rec_table`, `f_id`, таблицей значения и link target.
- Семантику каталоговых сущностей нельзя определять только по их ID.
- Regression-тест владения должен добавлять конфликтующую, но допустимую неизвестную строку и подтверждать её сохранение.
- Fixture-gate должен проверять все структурные разделы, заявленные в плане, а не только наличие базовых кодов.
