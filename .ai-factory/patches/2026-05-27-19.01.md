# Validate ATDB header before opening sql.js database

**Date:** 2026-05-27 19:01
**Files:** lib/sqlProcessor.ts
**Severity:** medium

## Problem

The refactored `parseAtdb` opened the input with `sql.js` before running the project-level SQLite header validation. Invalid or too-small `.atdb` buffers could therefore surface lower-level sql.js errors instead of the intended controlled validation errors.

## Root Cause

During the processor decomposition, buffer normalization and database creation were grouped together before the extracted `validateSqliteHeader` helper call. The original monolithic flow validated the header before opening the database, but that ordering was not preserved when turning `sqlProcessor.ts` into a thin facade.

## Solution

Moved `validateSqliteHeader(normalizedBuffer)` immediately after buffer normalization and before the dynamic `createDbFromBuffer` import and database creation. This restores the original validation boundary without changing the public `parseAtdb` API or logging behavior.

## Prevention

- During facade refactors, preserve validation and resource-acquisition order explicitly.
- Add a focused regression check for invalid `.atdb` buffers to ensure controlled errors are thrown before sql.js opens the file.
- Review parse/build changes for lifecycle boundaries: validate input first, acquire resources second, release resources in `finally`.
