# Корректный выбор основного участника life-event

**Date:** 2026-06-04 22:16
**Files:** lib/atdb/schemaContext.ts, lib/atdb/readers/eventsReader.ts, lib/atdb/readers/personsReader.ts, lib/atdb/writers/lifeEventWriter.ts, lib/atdb/writers/valueWriter.ts, scripts/check-atdb-write-safety.mjs
**Severity:** high

## Problem

Чтение и запись дат и мест рождения/смерти выбирали событие только по `EventRoles.et_id`. Из-за этого отец или мать могли получить данные события рождения ребенка, а запись места могла изменить чужое событие. Удаление места также пропускалось и оставляло старую ссылку.

## Root Cause

`EventDetails` хранит всех участников события, но код не проверял, что персона связана с событием через единственную основную роль. Общий helper записи ссылок также определял принадлежащий `vlink_table` только из нового значения, поэтому не мог выполнить точечное удаление при `null`.

## Solution

Для рождения чтение и запись теперь используют каталогизированную семантическую роль `bornPerson`, а для смерти — кэшируемый resolver единственной основной роли по `EventRoles.ismain`. Точечное удаление `ValuesLinks` принимает подтвержденный target-код отдельно от нового значения. Проверка безопасности записи дополнена сценариями второстепенного участника, изменения и удаления места.

При чтении ссылок подтвержденное write-safe поле получает приоритет над альтернативным fixture-specific полем, чтобы неизмененная пересборка не заменяла подтвержденное значение альтернативным.

## Prevention

- При чтении персональных данных из общего события всегда проверять семантическую роль участника.
- Проверять write-safe операции не только на неизмененной пересборке, но и на фактическом изменении и удалении значения.
- Для удаления полиморфной ссылки отдельно передавать подтвержденный target-код.
