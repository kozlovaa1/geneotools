# Блокировка записи дат с non-editable ValuesDates metadata

**Date:** 2026-07-02 18:01
**Files:** `lib/atdb/dates.ts`, `components/atdb-table/EventTable.tsx`, `scripts/check-atdb-dates.mjs`, `scripts/check-atdb-rebuild-contract.mjs`, `scripts/check-atdb-table-components.mjs`
**Severity:** high

## Problem

Review found a data-loss path in the new life-event date editing contract. A `ValuesDates` row with primary `y/m/d` plus additional metadata such as `calendar`, `sort*`, `lconf`, or `ltrust` could be treated as a simple editable date. The existing writer then deleted the row and reinserted only `y, m, d`, which would drop the metadata.

## Root Cause

`isSimpleAtdbDateMetadata` only checked `type` and secondary date parts. The reader already loaded additional `ValuesDates` metadata, but the simple/non-simple classifier did not include those fields, so preflight and UI editability made a narrower decision than the writer safety contract required.

## Solution

The simple-date classifier now treats any present `calendar*`, `sort*`, `lconf`, or `ltrust` value as non-editable metadata. That keeps such dates read-only for the simple date editor and causes strict rebuild preflight to block manual change-sets before the writer can replace the row. Tests now cover metadata classification and rebuild blocking. The event table also uses `dateInfo.display` so read-only non-simple dates keep their formatted display.

## Prevention

- When adding writable support for `ValuesDates`, compare the classifier against every metadata field loaded by the reader.
- Review generic `replaceOwnedValue` usage carefully for value tables with extra columns.
- Add both fixture-free metadata classification tests and rebuild/preflight tests for every newly blocked metadata shape.
