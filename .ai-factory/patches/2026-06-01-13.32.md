# Split generic ATDB schema check from yaman golden validation

**Date:** 2026-06-01 13:32
**Files:** scripts/inspect-atdb-schema.mjs, package.json, README.md
**Severity:** medium

## Problem

`scripts/inspect-atdb-schema.mjs` accepted alternate fixtures through `ATDB_SCHEMA_FIXTURE` or a positional argument, but `--check` always asserted exact `yaman-test.atdb` row counts and mappings. A valid different ATDB fixture could therefore fail the check even when inspection succeeded.

## Root Cause

The script mixed two responsibilities in one `--check` mode: generic structural validation for any inspected ATDB file and golden validation for the tracked `yaman-test.atdb` research fixture. The CLI surface suggested fixture flexibility, but the assertion logic assumed one specific database.

## Solution

Added generic snapshot validation for `--check` and isolated yaman-specific count/mapping assertions behind `--check-yaman`, while still applying yaman assertions automatically for the default fixture. Added an explicit `schema:atdb:check:yaman` npm script and documented the difference between generic and golden checks.

## Prevention

- Keep fixture-specific golden assertions separate from generic CLI validation.
- In review, check whether env/CLI override paths are compatible with hardcoded expectations.
- Verify generic tooling with a small synthetic fixture whose counts intentionally differ from the primary test fixture.
