# Рассинхронизация deferred query и visible selection

**Date:** 2026-06-24 14:55
**Files:** app/page.tsx, components/ScrollableDataTable.tsx, components/DataTable.tsx, components/atdb-table/AtdbTablePrimitives.tsx, scripts/check-atdb-table-components.mjs
**Severity:** high

## Problem

После добавления `useDeferredValue` table result мог временно строиться по устаревшему query snapshot, а toolbar, selection и bulk edit уже работали с текущим query state. В этом промежутке пользователь мог выбрать видимые строки или открыть массовое редактирование против stale `visibleIds`.

## Root Cause

Deferred state применялся к query активной вкладки как к отдельному объекту. При изменении поиска или фильтра rows и `visibleIds` отставали от toolbar state; при смене вкладки также существовал риск применить query старой сущности к новой активной сущности. Selection controls не знали, что rendered rows еще не соответствуют текущему query.

## Solution

Deferred snapshot перенесён на всю карту `tableQueries`, а rendered query выбирается из неё по текущей `activeEntity`. Это сохраняет совпадение сущности и query result. Пока deferred snapshot не догнал текущий state, selection checkbox, select-all, clear selection и bulk edit блокируются, чтобы stale `visibleIds` не участвовали в пользовательских действиях. Component regression script дополнен проверками этого контракта.

## Prevention

- Для deferred table state хранить единый rendered snapshot, а не смешивать immediate controls с stale visible rows без явной блокировки действий.
- Во время review проверять все действия, использующие `visibleIds`: select-all, deselect visible rows, visible selected counts и bulk entrypoints.
- Source-level regression должен фиксировать disabled state selection controls при pending table refresh.
