# Gender clear mismatch in strict ATDB rebuild

**Date:** 2026-06-22 19:46
**Files:** `lib/atdb/rebuildValidation.ts`, `scripts/check-atdb-rebuild-contract.mjs`, `docs/atdb_format.md`
**Severity:** medium

## Problem

An explicit `AtdbChangeSet` with `Person.gender` set to `null` or `undefined` passed preflight validation, but strict rebuild failed during post-build verification with `postbuild.change_not_visible`.

## Root Cause

The rebuild contract allowed `null` and `undefined` as clear values for write-safe fields, but `Person.gender` is not stored as a removable `Values*` row. It is stored in `Persons.sex`, where the existing normalized empty value is `0`, which reparses as `Unknown`. Post-build verification compared the rebuilt `Unknown` value to the literal expected `null` or `undefined`.

## Solution

Post-build validation now treats `gender: null` and `gender: undefined` as an expected `Unknown` value. Preflight also explicitly rejects runtime `changeSet` entity types outside `person`, `family`, and `place` before table routing can treat malformed JavaScript input as a place update. The rebuild contract harness now covers gender clearing and invalid entity type rejection.

## Prevention

- Review scalar fields separately from removable `Values*` rows when documenting clear semantics.
- Add regression coverage for every documented `null` and `undefined` behavior in public change-set APIs.
- Validate runtime shape at public API boundaries even when TypeScript types already constrain internal callers.
