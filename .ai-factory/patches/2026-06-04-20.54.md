# Диагностика выбора ATDB mapping и синхронизация документации

**Date:** 2026-06-04 20:54
**Files:** lib/atdb/mapping.ts, lib/atdb/schemaContext.ts, lib/atdb/readers/personsReader.ts, scripts/check-atdb-write-safety.mjs, docs/atdb_schema_yaman.md, AGENTS.md
**Severity:** medium

## Problem

Resolver возвращал разрешённые mapping-правила без безопасной диагностики выбранного уровня доверия. Использование legacy fallback для семантики родительских ролей не создавало обязательный `WARN`. Публичная документация сохраняла устаревшие значения parse-build drift и представляла ID ролей как универсальные.

## Root Cause

Диагностика была добавлена для итоговой сводки схемы и ошибок разрешения, но успешный путь resolver остался без событий. Родительские роли разрешались по каталогу корректно, однако legacy-правила использовались напрямую в reader, минуя единый диагностируемый resolver. Документация не была полностью синхронизирована после изменения write-side поведения.

## Solution

- Добавлены безопасные события `mapping.rule.resolved` и `mapping.legacy-fallback.used`.
- Добавлен `resolveMappedEventRole`, который связывает legacy-правило с фактическим каталогом `EventRoles`.
- Решения resolver кэшируются внутри `AtdbSchemaContext`, чтобы не повторять одинаковые диагностики для каждой сущности.
- Regression-проверка подтверждает наличие invariant и legacy-fallback диагностик.
- Устаревшие утверждения о drift и универсальных ID ролей исправлены.

## Prevention

- Каждый публичный resolver mapping должен диагностировать успешный выбор правила и его confidence.
- Legacy fallback должен применяться только через единый resolver с обязательным `WARN`.
- После изменения smoke-результатов необходимо искать старые числовые ожидания во всех публичных документах.
