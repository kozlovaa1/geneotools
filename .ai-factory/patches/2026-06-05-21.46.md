# Idempotent write-safe Values replacement

**Date:** 2026-06-05 21:46
**Files:** lib/atdb/writers/valueWriter.ts
**Severity:** high

## Problem

`npm run test:atdb:write-safety` failed with `unchanged owned event places changed`.
The parse/build flow rewrote owned `ValuesLinks` rows even when parsed data was
written back without user changes.

## Root Cause

`replaceOwnedValue` always executed `DELETE` followed by `INSERT` for every
write-safe value. For unchanged data this preserved semantic values but changed
SQLite rows, including generated row IDs. The write-safe contract requires
unknown and unchanged owned values to remain stable unless the user actually
changes the value.

## Solution

Added a read-before-write check inside `replaceOwnedValue`. The writer now reads
the currently owned rows for the exact `(valueTable, f_id, rec_table, rec_id)`
scope, including `vlink_table` for link values, and returns early when the
existing row already equals the requested value or when a null write targets an
empty scope.

The no-op path logs a safe diagnostic with `fix.value.write.noop`; it includes
only field ID and `rec_table`, not raw values or personal data.

## Prevention

- Review write-safe helpers for idempotency, not only semantic equality after
  reparse.
- Tests should compare unchanged owned rows before and after parse/build,
  including SQLite row identity when the contract says unchanged rows must stay
  untouched.
- Do not model `Families` as nuclear families; in ATDB6 this table represents
  "Роды". Links from persons to a род are a different relationship and must not
  be interpreted as husband/wife/children.
