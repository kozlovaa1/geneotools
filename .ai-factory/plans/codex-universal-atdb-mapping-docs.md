## Связь с дорожной картой

Milestone: «Универсальный mapping и документация формата ATDB»

Обоснование: план выделяет общий слой правил формата из результатов multi-fixture исследования и делает `docs/atdb_format.md` каноническим описанием, сохраняя fixture-документы как доказательства совместимости.

## Исследовательский контекст

Источник: `.ai-factory/RESEARCH.md` (`Active Summary`) и `docs/atdb_multi_fixture_schema.md`

Цель:
- Централизовать mapping `rec_table`, `Fields`, `EventTypes`, `EventRoles` и `Values*`, устранив противоречия между кодом и документацией.

Ограничения:
- SQL-логика остаётся в `lib/`; UI не работает со схемой SQLite напрямую.
- Публичные артефакты не содержат персональные строки, GUID, пути, заметки и raw `ValuesStr`.
- `Fields.area` не считается надёжным универсальным источником семантики.
- Fixture-specific наблюдения и legacy fallback нельзя повышать до invariant-правил без доказательств.

Решения:
- Разделить правила на `invariant`, `fixture-specific observation` и `legacy fallback`.
- Использовать `rec_table=7` для `Events`, `9` для `Families`, `13` для `Persons`, `14` для `Places` как подтверждённый общий baseline трёх fixtures.
- Сохранять `docs/atdb_multi_fixture_schema.md` как compatibility evidence и regression-контур, а не второй канон формата.

Открытые вопросы для реализации:
- Какие legacy fallback нужны только для чтения, а какие допустимы при записи.
- Следует ли runtime resolver отклонять неоднозначный mapping или возвращать диагностируемый fallback.

### Этап 1. Модель и разрешение правил

- [x] **Задача 1. Создать типизированный канонический реестр правил формата ATDB.**
  - Создать единый машиночитаемый источник правил, доступный одновременно TypeScript runtime и Node.js-проверочным скриптам без дублирования числовых кодов; предпочтительная форма — JSON-реестр с типизированной TypeScript-обёрткой и runtime-валидацией структуры.
  - Создать отдельный модуль mapping в `lib/atdb/` с типами для кодов таблиц, определений полей, таблиц `Values*`, link targets, типов и ролей событий.
  - Явно хранить для каждого правила уровень доверия: `invariant`, `fixture-specific` или `legacy-fallback`, а также область применения read/write.
  - Перенести подтверждённые числовые коды из `lib/atdb/constants.ts` и разбросанные field IDs из readers/writers; исключить конкурирующие значения для `Events`, `Persons` и `Families`.
  - Зафиксировать, что неизвестные и fixture-specific значения остаются доступными как данные, но не получают универсальную семантику автоматически.
  - Файлы: `lib/atdb/constants.ts`, новые `lib/atdb/mapping.json`, `lib/atdb/mapping.ts` и при необходимости `lib/atdb/mappingTypes.ts`.
  - Логирование: сам статический реестр не логирует; его публичные resolver-функции должны поддерживать безопасные `DEBUG`-сообщения о выбранном уровне правила и `WARN` о legacy fallback без значений пользовательских полей.

- [x] **Задача 2. Реализовать runtime resolver схемы и fallback-политику.** Зависит от задачи 1.

  - Создать единый `AtdbSchemaContext`, который разрешается один раз на открытую базу в `parseAtdb`/`buildAtdb` и передаётся readers/writers вместо повторного чтения каталогов и независимых решений.
  - Не использовать `Fields.area` как единственное доказательство семантики; учитывать `tablecode`, `datatype`, таблицу значения и подтверждённые link targets.

## Критерии завершения

- В коде существует один явный типизированный слой правил ATDB с уровнями `invariant`, `fixture-specific` и `legacy-fallback`.
- Runtime и Node.js mapping-gate используют один машиночитаемый источник правил без дублирования кодов.
- Resolver создаёт единый `AtdbSchemaContext` на открытую базу и передаёт безопасные диагностические события без прямого подробного вывода по умолчанию.
- Readers и writers не используют противоречивые значения `rec_table` для основных сущностей.
- Write-side применяет только явно разрешённые write-safe правила, не создаёт роли при неоднозначном mapping и не уничтожает неизвестные данные.
- Regression-проверки подтверждают сохранение неизвестных `Values*` и запрет небезопасной записи.
- Mapping-gate проверяет `yaman-test.atdb`, `yaman-test-full.atdb`, `family-test.atdb` через allow-list registry и соблюдает redaction.
- `docs/atdb_format.md` является каноническим справочником, а multi-fixture документ остаётся доказательством совместимости.
- Все обязательные проверки проходят; известный parse-build drift остаётся видимым warn-only сигналом для следующего milestone.
