# Базовые правила проекта GeneoTools

> Автоматически выявленные соглашения по текущей кодовой базе. При необходимости уточняйте этот файл вручную.

## Соглашения по именованию

- Файлы компонентов React: `PascalCase.tsx` (`FileUploader.tsx`, `ScrollableDataTable.tsx`).
- Файлы утилит и бизнес-логики: `camelCase.ts` (`sqlProcessor.ts`, `initSqlJs.ts`).
- Переменные и функции: `camelCase`.
- Компоненты, интерфейсы и типы: `PascalCase`.
- SQL-таблицы и поля сохраняют имена исходной `.atdb`/SQLite схемы.

## Структура модулей

- `app/` содержит страницы и корневой layout Next.js App Router.
- `components/` содержит клиентские React-компоненты и не должен напрямую выполнять SQL-запросы.
- `lib/` содержит работу с `.atdb`, `sql.js`, типы и общие утилиты.
- `docs/` содержит справочную документацию по формату данных и анализу кодовой базы.
- `public/` содержит статические изображения и логотипы.
- `scripts/` содержит вспомогательные проверки и сборочные скрипты.
- Docker-конфигурация хранится в `Dockerfile` и `docker-compose.yml`.

## Доменная модель ATDB

- `Families` в ATDB6 означает «Роды», а не нуклеарные семьи.
- Не выводить из `Families` роли «муж», «жена» и «дети»; родственные и супружеские связи определяются через события, `EventDetails` и `EventRoles`.
- Связь персоны с родом является отдельной ссылкой на род и не задаёт родительство, брак или состав семьи.

## Обработка ошибок

- Ошибки чтения и сборки `.atdb` обрабатываются через `try/catch`.
- Пользовательские ошибки отображаются в UI через React state.
- В `lib/` допустимо выбрасывать `Error` с понятным сообщением, а UI-слой должен преобразовывать его в пользовательский текст.

## Логирование

- Используется `console.error` для критичных ошибок пользовательского сценария.
- Используется `console.warn` для необязательных таблиц или данных, отсутствие которых не должно прерывать обработку.
- Не выводить содержимое пользовательских `.atdb` файлов, персональные данные или секреты в логи.

## Коммуникация

- Если пользователь пишет по-русски, отвечать естественным русским языком и так же писать русскоязычную документацию: переводить общеупотребимые технические слова и фразы, не смешивать русский с английским без необходимости. Английские названия оставлять только для имён API, команд, путей, библиотек, типов, таблиц и терминов без точного русского аналога.

## Тестирование и проверка

- Основная быстрая проверка: `npm run lint`.
- Для изменений в парсинге `.atdb` запускать `npm run smoke:atdb`, если тестовая база доступна локально.
- Перед изменениями парсинга или сборки `.atdb` дополнительно проверять сценарии загрузки и экспорта тестовой базы.
- Сохранять строгую типизацию TypeScript и избегать `any`, если можно выразить тип через существующие модели из `lib/types.ts`.
