# Анализ кодовой базы GeneoTools (2026-02)

## Методика анализа

Проверка выполнена по слоям приложения:

1. UI-слой (`app/`, `components/`)
2. Слой обработки данных (`lib/`)
3. Качество кода (ESLint, структура, дублирование)
4. Актуальность документации

Использованные команды:
- `rg --files`
- `npm run lint`
- `rg -n "TODO|FIXME|HACK|console\.log" app components lib *.ts *.js`

## Краткая архитектурная картина

- **Точка входа UI:** `app/page.tsx`
- **Основной workflow:** загрузка `.atdb` → `parseAtdb` → отображение таблиц → `buildAtdb` → скачивание
- **Ядро доменной логики:** `lib/sqlProcessor.ts`
- **Рендер данных:** `components/ScrollableDataTable.tsx` + `components/DataTable.tsx`

## Сильные стороны

- Приложение полностью работает локально в браузере, без серверного хранения данных.
- В `sqlProcessor` заложена гибкая логика для разных вариантов схемы (`PRAGMA table_info`, fallback-мэппинг полей).
- Есть обработка метаданных (`Global`) и поддержка нескольких сущностей (`persons`, `families`, `events`, `places`).
- UI простой и понятный для MVP-сценария.

## Выявленные проблемы

### 1) Высокая связность и перегруженность `lib/sqlProcessor.ts`

- Один большой модуль совмещает:
  - валидацию формата файла,
  - извлечение метаданных,
  - парсинг разных сущностей,
  - эвристики интерпретации полей,
  - построение выходной структуры.
- Это осложняет тестирование, ревью и безопасные изменения.

### 2) Дублирование типов

- Интерфейсы `Person`, `Family`, `Event`, `Place` повторяются в нескольких файлах (`lib/parseAtdb.ts`, `lib/sqlProcessor.ts`, `components/DataTable.tsx`).
- Риск расхождения контрактов между UI и парсером.

### 3) Проблемы линтинга и технический долг

`npm run lint` возвращает ошибки и предупреждения:

- **Ошибки:**
  - `no-explicit-any` в таблицах/компонентах
  - `prefer-const`
  - `no-require-imports` в тестовых скриптах
- **Предупреждения:**
  - неиспользуемые переменные/импорты
  - пропущенные зависимости `useCallback`
  - debug-логирование в UI

Следствие: нестабильное качество кода и усложнение CI в будущем.

### 4) Непоследовательность структуры UI-слоя

- В `app/page.tsx` есть неиспользуемые импорты и переменные.
- `ScrollableDataTable` содержит неиспользуемые части (переменные/обработчики), что указывает на незавершённую эволюцию компонента.
- `DataTable` содержит большой объем логики сортировки и рендера в одном файле.

### 5) Устаревшие/несогласованные документы

- В README и DOCS частично устаревшая структура проекта (например, не все текущие компоненты отражены).
- Некоторые формулировки про «парсинг на сервере» не соответствуют текущей клиентской реализации.

## Риски при развитии без рефакторинга

1. Рост числа регрессий при изменениях в парсере.
2. Усложнение внедрения новых таблиц/полей формата `.atdb`.
3. Рост стоимости поддержки UI-слоя.
4. Блокировка автоматизации качества из-за текущих lint-error.

## Рекомендации (высокий приоритет)

1. Нормализовать доменные типы в единый модуль (`lib/types.ts`).
2. Разбить `sqlProcessor.ts` на подмодули (metadata/persons/families/events/places + mappers).
3. Довести `npm run lint` до green-статуса.
4. Удалить debug-логи из прод-рендера.
5. Обновить документацию по фактической архитектуре и ограничениям.
