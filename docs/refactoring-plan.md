# План рефакторинга GeneoTools

## Цели

- Снизить связанность кода и упростить сопровождение.
- Выровнять контракты типов между UI и data-слоем.
- Подготовить проект к безопасному расширению парсинга `.atdb`.
- Привести статический анализ (`eslint`) к стабильному green.

## Принципы

1. **Малые инкременты** — каждый этап должен быть изолирован и проверяем.
2. **Сначала контракты, потом реализация** — сперва унифицируем типы.
3. **Без изменения поведения на ранних этапах** — сначала структурный рефакторинг.
4. **Измеримость** — на каждом этапе фиксируем результат (lint/build/ручной smoke).

## Этапы

### Этап 0 — Базовая стабилизация качества (1 день)

**Задачи:**
- Исправить текущие lint-errors (`no-explicit-any`, `prefer-const`, `no-require-imports`).
- Удалить неиспользуемые импорты/переменные.
- Убрать debug-логирование или обернуть его в dev-flag.

**Критерий готовности:**
- `npm run lint` без ошибок.

---

### Этап 1 — Унификация доменной модели (1 день)

**Задачи:**
- Создать `lib/types.ts` как единственный источник доменных интерфейсов:
  - `Person`, `Family`, `Event`, `Place`, `ParsedAtdb`, `Metadata`.
- Перевести `DataTable`, `ScrollableDataTable`, `app/page.tsx`, `lib/*` на импорт из `lib/types.ts`.
- Удалить дубли интерфейсов в других файлах.

**Критерий готовности:**
- Нет повторяющихся доменных интерфейсов в проекте.

---

### Этап 2 — Декомпозиция `sqlProcessor` (2–3 дня)

**Предлагаемая целевая структура:**

```text
lib/
  sql/
    parse/
      index.ts
      parseMetadata.ts
      parsePersons.ts
      parseFamilies.ts
      parseEvents.ts
      parsePlaces.ts
    build/
      index.ts
    mapping/
      fieldDefinitions.ts
      eventRoleMapping.ts
    db.ts
```

**Задачи:**
- Выделить повторно используемые SQL-хелперы (safe query, stmt lifecycle).
- Вынести parsing-пайплайн по сущностям в отдельные файлы.
- Минимизировать side-effects и вложенные условные блоки.

**Критерий готовности:**
- `sqlProcessor` становится фасадом/точкой входа, а не монолитом.

---

### Этап 3 — Упрощение табличного UI (1–2 дня)

**Задачи:**
- Вынести общую сортировку в `lib/table/sort.ts`.
- Декомпозировать `DataTable` на подкомпоненты:
  - `PersonsTable`, `FamiliesTable`, `EventsTable`, `PlacesTable`.
- Удалить мертвый код в `ScrollableDataTable` (неиспользуемые refs/handlers/state).

**Критерий готовности:**
- Размер и когнитивная сложность компонентов заметно снижены.

---

### Этап 4 — Тестовый контур (2 дня)

**Задачи:**
- Добавить unit-тесты на маппинг ключевых полей (`ValuesStr`, `ValuesDates`, `EventDetails`).
- Добавить smoke-тест парсинга эталонного `.atdb`.
- Перевести `test-parsing.js/ts` на единый формат (ESM/TS), совместимый с линтером.

**Критерий готовности:**
- Критичные ветви парсинга покрыты тестами.

---

### Этап 5 — Документация и DX (0.5–1 день)

**Задачи:**
- Обновить README и DOCS под фактическую структуру.
- Добавить раздел «ограничения и известные допущения».
- Зафиксировать в документации контракт версии `.atdb`, поддерживаемой в MVP.

**Критерий готовности:**
- Документация синхронизирована с реальным кодом.

## Приоритизация задач (MoSCoW)

- **Must:** этапы 0–2
- **Should:** этап 3
- **Could:** этап 4
- **Won’t (пока):** оптимизация производительности для очень больших БД (>100k персон)

## Риски и смягчение

1. **Риск:** случайное изменение поведения парсинга при декомпозиции.  
   **Смягчение:** snapshot/smoke-сравнение результата до/после.

2. **Риск:** расхождение в интерпретации полей `f_id` для разных баз.  
   **Смягчение:** централизованный слой mapping + лог fallback-правил.

3. **Риск:** рост времени на ручную проверку UI.  
   **Смягчение:** минимальный чек-лист smoke-проверок после каждого этапа.

## Definition of Done для полного рефакторинга

- `npm run lint` проходит без ошибок.
- Доменные типы не дублируются.
- Парсер разделён на небольшие модули с понятной ответственностью.
- Есть минимальный набор автотестов на критичный parsing-flow.
- README/DOCS соответствуют реальной архитектуре.
